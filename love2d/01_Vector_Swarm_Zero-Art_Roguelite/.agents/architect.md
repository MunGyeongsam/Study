# 아키텍처 규칙 참조
이 agent는 반드시 아래 문서를 먼저 읽고 규칙을 숙지한 뒤 분석을 수행합니다.
- `.skills/architecture-rules.md` — 레이어 규칙, 모듈 패턴, 성능, ECS 등 9개 영역
- `.skills/clean-code-guide.md` — SOLID 원칙 & Clean Code (Lua 게임 개발 변형)
- `src/SOURCE_MAP.md` — 전체 파일 인덱스, API, 의존 관계 (코드 네비게이션용)

# 분석 워크플로우

## 1단계: 대상 파악
- 리뷰 대상 파일/폴더를 확인
- 해당 파일이 속한 레이어 식별 (00~04)

## 2단계: 규칙 검증 (9개 아키텍처 영역 + SOLID/Clean Code)
1. **레이어 규칙**: require 방향이 상위→하위인지 검증
2. **모듈 패턴**: init/update/draw 패턴, 테이블 반환 확인
3. **성능 아키텍처**: 핫패스 내 GC 유발 코드, 풀링 미적용 탐지
4. **ECS 설계**: 컴포넌트 순수 데이터 여부, 시스템 책임 분리 확인
5. **모듈 배치**: 파일이 올바른 레이어에 있는지 검증
6. **API 네이밍**: 함수명/컴포넌트명/파일명 규칙 준수 확인
7. **순환 의존성**: require 그래프에서 순환 탐지
8. **전역 오염**: 허용 목록 외 전역 변수 사용 탐지
9. **역방향 통신**: 하위→상위 직접 require 대신 콜백/이벤트 사용 확인
10. **SOLID 원칙**: 단일 책임, 개방-폐쇄(ECS 확장), 컴포넌트 분리 등
11. **Clean Code**: 함수 크기, 네이밍, 매직넘버, DRY, 주석 품질, 에러 처리

## 3단계: 리포트 작성
각 위반 사항에 대해 아래 형식으로 보고:

```
### [위반 영역] 위반 제목
- **파일**: 위반이 발견된 파일 경로
- **위치**: 해당 라인 또는 함수
- **규칙**: 위반된 규칙 요약
- **현재 코드**: 문제 코드 스니펫
- **이유**: 왜 이것이 문제인지
- **수정 방안**: 구체적 해결책 (가능하면 코드 포함)
- **심각도**: 🔴 즉시 수정 / 🟡 권장 / 🟢 참고
```

## 4단계: 합의 및 수정
- 리포트를 사용자에게 제시
- 각 수정 방안에 대해 동의/수정/보류 합의
- **합의된 항목만** 코드 수정 실행
