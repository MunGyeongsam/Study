# Vector Swarm — Planning Guide

---

## 1. 게임 디자인 원칙 (Design Principles)

모든 기획은 아래 원칙을 **우선순위 순서대로** 적용한다.

### 1.1 Fun-First (재미 우선)

> 기술적으로 뛰어나도 재미없으면 삭제한다.

**4가지 재미 기둥** — 신규 기능은 최소 1개에 기여해야 한다:

| 기둥 | 질문 | 이 게임에서의 표현 |
|------|------|-------------------|
| 긴장감 (Tension) | "죽을 수도 있다" | 탄막 사이 회피 |
| 성취감 (Mastery) | "내가 성장했다" | 빌드 완성, 기록 갱신 |
| 놀라움 (Wonder) | "이것도 되네!" | 스킬 시너지, 새 패턴 |
| 몰입감 (Flow) | "한 판만 더" | 짧은 세션, 리듬감 |

**검증 체크리스트** (기능 설계 시 반드시 확인):
- [ ] 4기둥 중 최소 1개에 기여하는가?
- [ ] 즉각적 피드백이 있는가?
- [ ] 이 기능 없으면 게임이 덜 재미있는가?
- [ ] 다른 재미 요소를 방해하지 않는가?
- [ ] 처음 30초 안에 효과를 느낄 수 있는가?

### 1.2 15-Minute Run (세션 타겟)

- 1회 플레이: **10~15분**
- 스테이지 당: **30~90초**
- 최대 스테이지: ~10개 (Early Access 기준)
- "한 판 더" 욕구를 유발하는 길이 = 통학/출퇴근 1회에 2~3판

### 1.3 MVP 우선 (Minimum Viable Play)

- 핵심 루프를 먼저 완성한 뒤 확장
- 기획 문서에 "확장 가능" 표시만 하고 **당장 구현하지 않는다**
- 과도한 추상화/시스템 금지: 1회 사용 코드에 팩토리 패턴 불필요
- 빈 stub API로 인터페이스만 확보하고 내용은 나중에 채우기 OK

### 1.4 Feel Before Balance

- 숫자 밸런싱보다 **조작감/타격감** 먼저
- "대충 강해 보이는 게" 초기엔 밸런스보다 중요
- 플레이 테스트로 체감 → 숫자 조정 순서

## 2. ECS 기반 기능 분해 (Feature Decomposition)

새 기능을 기획할 때, ECS 단위로 분해하여 산출물을 명확히 한다.

### 2.1 분해 템플릿

```
## 기능: [기능명]

### 새 컴포넌트 (03_game/components/)
- ComponentName: { field1, field2, ... }  -- 순수 데이터만

### 새 시스템 (03_game/systems/)
- systemName: requires [Comp1, Comp2] → 매 프레임 동작 설명

### 팩토리 수정 (03_game/entities/entityFactory.lua)
- createXxx(params): 엔티티 생성 + 컴포넌트 부착

### 기존 시스템 수정
- existingSystem.lua: 변경 내용 설명

### HUD/UI 변경 (04_ui/)
- 표시할 정보, 조작 변경

### 성능 고려사항
- 빈번 생성/파괴 → 풀링 필요 여부 (bulletPool 패턴 참조)
- 매 프레임 실행 → 핫패스 최적화 필요 여부
- GC 유발 패턴 (테이블 생성, 문자열 연결) 회피 방안

### 레이어 의존성 확인
- 새 모듈이 require하는 대상 (상위→하위만 허용)
```

### 2.2 분해 전 확인

기능 분해 전에 기존 자산을 반드시 확인한다:
- `src/03_game/components/` — 이미 존재하는 컴포넌트 확인 (재사용 우선)
- `src/03_game/systems/` — 기존 시스템에 로직 추가 가능한지 검토
- `src/03_game/entities/entityFactory.lua` — 기존 팩토리 함수 확인
- 새로 만들기 전에 "기존 것을 확장할 수 있는가?" 질문

### 2.3 분해 원칙

| 원칙 | 설명 |
|------|------|
| **컴포넌트 = 순수 데이터** | 로직 없음. `{ name, defaults, new() }` 패턴 |
| **시스템 = 단일 책임** | 1시스템 = 1관심사. 쿼리 기반 엔티티 처리 |
| **팩토리 = 조립** | 컴포넌트 조합으로 엔티티 생성 |
| **핫패스 예외** | 성능 임계 시스템은 인라이닝 허용 (주석 필수) |
| **풀링 우선** | 빈번 생성/파괴 객체는 반드시 풀 사용 (bulletPool 패턴) |

### 2.4 컴포넌트 네이밍 규칙

- PascalCase: `Transform`, `BulletEmitter`, `EnemyAI`
- 파일명은 camelCase: `transform.lua`, `bulletEmitter.lua`
- `Tag` 접미사: 마커 컴포넌트 (`PlayerTag`, `EnemyTag`)
- `State` 접미사: 상태 데이터 (`DashState`, `FocusState`)

## 3. Phase 기반 개발 프로세스

### 3.1 Phase 구조

```
Phase 0: 기반 구축          ✅ (엔진, ECS, 카메라, UI 레이아웃)
Phase 1: MVP 게임 루프      ✅ (이동→적→탄막→피격→사망→재시작)
Phase 2: 게임플레이 확장    🔄 (공격, 대쉬, 레벨업, 스테이지)
Phase 3: 콘텐츠 & 비주얼   🔲 (보스, 적 다양성, 이펙트, 사운드)
Phase 4: 폴리싱 & 출시      🔲 (메뉴, 메타성장, 최적화, 포팅)
```

### 3.2 Phase 내 작업 단위

- **Sub-phase**: 2A, 2B, 2C, 2D... (독립 기능 묶음)
- **Task**: 2A.1, 2A.2... (커밋 가능한 최소 단위)
- 1 Task = **1~2시간** 작업 분량 목표
- 각 Sub-phase 완료 후 **게임 실행 테스트 필수**

### 3.3 Sub-phase 계획 순서

1. **목표 정의**: 이 Sub-phase가 끝나면 플레이어가 뭘 할 수 있나?
2. **선택지 비교**: 구현 방식 A/B/C 장단점 비교 (최소 2안)
3. **ECS 분해**: 위 2.1 템플릿으로 산출물 명확화
4. **의존성 분석**: 기존 코드 중 수정이 필요한 부분 파악
5. **작업 순서**: 의존성 방향으로 Task 순서 결정
6. **위험 요소**: 기술적 불확실성, 성능 우려, 밸런스 민감도
7. **검증 기준**: "이걸 확인하면 완료" 조건 명시

### 3.4 계획서 출력 형식

```markdown
# [Sub-phase ID]: [제목]

## 목표
> 한 줄 목표 + 재미 기둥 연결

## 설계 결정
| 선택지 | 장점 | 단점 | 판정 |
|--------|------|------|------|
| A: ... | ... | ... | ✅/❌ |

## ECS 분해
(2.1 템플릿 사용)

## 작업 목록
| # | Task | 산출물 | 의존성 |
|---|------|--------|--------|
| 1 | ... | file.lua 생성 | 없음 |
| 2 | ... | system.lua 수정 | Task 1 |

## 위험 요소
- ⚠️ [위험]: [대응 방안]

## 검증 기준
- [ ] 게임 실행 시 [조건] 확인
- [ ] [수치] 범위 내 동작
```

## 4. 난이도 & 밸런스 설계 프레임워크

### 4.1 난이도 곡선 원칙

```
난이도
  ▲
  │        ╱╲    ╱╲     ╱╲
  │      ╱    ╲╱    ╲  ╱    ╲   ← 웨이브 내 긴장/이완 반복
  │    ╱                 ╲
  │  ╱                     ╲
  │╱                         ╲
  └──────────────────────────── ▶ 시간/스테이지
      쉬움    중간    어려움    보스
```

- **점진적 상승**: 스테이지가 진행될수록 기본 난이도 증가
- **파동 패턴**: 각 웨이브/스테이지 내에서 긴장↔이완 반복
- **브레이크 타임**: 스테이지 사이에 짧은 안전 구간 (레벨업 선택 등)

### 4.2 난이도 파라미터

| 파라미터 | 초기값 | 스케일링 방식 |
|----------|--------|--------------|
| 웨이브 당 적 수 | 3~5 | +1~2 per stage |
| 적 이동 속도 | 1.0x | +0.1x per stage (cap 2.0x) |
| 적 HP | 1.0x | +0.15x per stage |
| 탄막 밀도 | 기본 | 패턴 복잡도 증가 |
| 탄막 속도 | 1.0x | +0.05x per stage (cap 1.5x) |
| 동시 적 수 상한 | 8 | +2 per stage (cap 20) |

### 4.3 밸런스 설계 순서

1. **체감 먼저**: 대략적 수치로 플레이, "느낌" 확인
2. **DPS 계산**: 플레이어 공격력 vs 적 HP → 처치 시간 목표 (2~5초)
3. **생존 시간**: 평균 플레이어가 스테이지 N까지 도달하는 시간 체크
4. **레벨업 보상**: 레벨업 선택이 "체감될 만큼" 강한지 확인
5. **수치 조정**: 위 결과에 따라 ±20% 범위에서 미세 조정

### 4.4 밸런스 레드라인 (절대 규칙)

| 규칙 | 이유 |
|------|------|
| 즉사 금지 (첫 3스테이지) | 초보 이탈 방지 |
| 무적 빌드 불가 | 긴장감 유지 |
| 1초 이내 화면 가득 탄막 금지 | 반응 시간 보장 |
| 레벨업 선택지 중 "함정" 없음 | 모든 선택이 의미 있어야 함 |
| 최소 2가지 회피 경로 존재 | 패턴 암기 강요 방지 |

### 4.5 적 자동 디스폰

마지막 1~2마리가 도망가서 진행이 막히는 걸 방지:
- 적 생존 시간 제한: **~15초** (스폰 후)
- 디스폰 전 **2초간 깜빡임** 경고
- 디스폰된 적은 XP 미지급 (킬과 구분)
- 웨이브 클리어 판정: 모든 적 dead **또는** despawned
