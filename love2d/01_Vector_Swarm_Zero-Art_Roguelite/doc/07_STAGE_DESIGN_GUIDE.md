# 07. 스테이지 디자인 가이드

> 스테이지 진행 시스템의 설계 방향과 확장 계획을 정리하는 문서.
> 마지막 갱신: 2026-04-22

---

## 1. 설계 철학

### 핵심 원칙
- **점진적 긴장**: 스테이지가 올라갈수록 자연스럽게 위험도 증가
- **무한 확장**: 정의된 스테이지 이후는 규칙으로 자동 생성 (데이터 없어도 동작)
- **규칙 우선**: 개별 스테이지 데이터보다 범용 규칙을 선호 (제약의 미학)
- **확장 슬롯**: 보스, 변형, 포메이션, 테마 등 필드 추가만 하면 됨

### 현재 구조 (Phase 5A)

> **데이터 분리**: 스테이지 정의는 `stageData.lua`, 포메이션은 `formationDefs.lua`에 순수 데이터로 분리.
> 스폰 로직은 `stageManager.lua`에서 `stageData`/`formationDefs`를 require하여 사용.

```
Stage 1~5  → 손작업 STAGE_DEFS (적 풀, 스폰 방향 고정)
Stage 3,6,9,12,15 → 보스 스테이지 (보스 1체 처치 = 클리어)
Stage 6~14 (비보스) → 테마 규칙 자동 생성 (§2.1)
Stage 16+  → 동일 규칙 무한 연속 + 매 3스테이지 보스
```

## 2. 스테이지 정의 테이블

### 수동 디자인 스테이지 (1~5)

| Stage | Waves | 스폰 방향 | 적 타입 풀 | 의도 |
|-------|-------|----------|-----------|------|
| 1 | 3 | 위 100% | bit, node | 입문. 위만 보면 OK |
| 2 | 4 | 위 80%, 좌우 각 10% | bit, node, vector | 좌우 인지 시작 |
| 3 | — | (보스: NULL) | — | 첫 보스전 |
| 4 | 5 | 위 50%, 좌우 각 20%, 아래 10% | bit, node, vector, loop | 풀 라인업 + **swift 보장** |
| 5 | 6 | 위 40%, 좌우 각 20%, 아래 20% | node, vector, loop, matrix | 고난이도 진입 |

### 자동 생성 스테이지 (6+)

Stage N (N ≥ 6, 비보스) 일 때:
- **Waves**: `min(5 + floor((N-5)/2), 8)` → 최대 8웨이브
- **스폰 방향**: 위 40%, 좌우 각 20%, 아래 20% (Stage 5와 동일, 고정)
- **적 타입**: 테마 규칙에 따라 결정 (아래 §2.1)
- **변형 등장**: 변형 규칙에 따라 결정 (§2.1 규칙 2~3)
- **난이도 수식**: 기존 공식 유지 (cap 적용)

> **적용 범위**: Stage 1~5는 손작업 STAGE_DEFS 그대로 사용. 테마 규칙은 **6+ 자동 생성에만 적용**.

### 2.1 스테이지 테마 규칙 (5A.3f)

> **설계 원칙**: 데이터 15개를 손으로 채우는 대신, **4가지 규칙으로 무한 스테이지의 다양성을 자동 생성**.
> 이것이 제약의 미학(Constraint Beauty) — 코드만으로 스테이지 개성을 만든다.
>
> **적용 범위**: Stage 1~5는 손작업 STAGE_DEFS 유지. 아래 규칙은 **Stage 6+ 자동 생성에만 적용**.

#### 규칙 1: 스테이지 테마 (적 풀 결정)

6+ 비보스 스테이지는 **보스를 건너뛴 순번**으로 기동형/화력형이 교대:

| 비보스 순번 | 실제 스테이지 | 테마 | 플레이어 체감 |
|:-----------:|:----------:|------|-------------|
| 1번째 | Stage 1 | (손작업) | 입문 |
| 2번째 | Stage 2 | (손작업) | 좌우 인지 |
| — | Stage 3 | **보스 NULL** | — |
| 3번째 | Stage 4 | (손작업) | swift 첫 등장 |
| 4번째 | Stage 5 | (손작업) | 고난이도 |
| — | Stage 6 | **보스 STACK** | — |
| 5번째 | Stage 7 | **기동형** | "빠르다! 피해!" + armored 첫 등장 |
| 6번째 | Stage 8 | **화력형** | "탄막 숲!" |
| — | Stage 9 | **보스 HEAP** | — |
| 7번째 | Stage 10 | **기동형** | splitter 첫 등장 |
| 8번째 | Stage 11 | **화력형** | 탄막 밀도 최고조 |
| — | Stage 12 | **보스 RECURSION** | — |
| 9번째 | Stage 13 | **기동형** | shielded 첫 등장 |
| 10번째 | Stage 14 | **화력형** | 최종 화력 테스트 |
| — | Stage 15 | **보스 OVERFLOW** | 최종 보스 |
| 11번째 | Stage 16 | **기동형** | Endless 진입 |
| ... | ... | 교대 반복 | ... |

테마별 적 풀:
```lua
MOBILITY_POOL  = {"bit", "vector", "loop"}   -- 이동으로 위협
FIREPOWER_POOL = {"node", "loop", "matrix"}  -- 탄막으로 위협
-- loop: 궤도(이동) + 나선탄(화력) = 양쪽 소속

-- 적 선택 시: 테마 풀에서 70%, 전체 풀에서 30% (단조로움 방지)
```

**왜 2가지 테마인가?**
- 기동형: 대쉬·이동 스킬 테스트 → "반응 속도" 긴장감
- 화력형: 포커스·패턴 읽기 테스트 → "집중" 긴장감
- 교대 배치로 긴장의 질이 스테이지마다 바뀜 → **몰입 유지**

#### 규칙 2: 변형 보장 등장 (Guaranteed Encounter)

> **쾌감(Juice) 보장**: 새 변형의 "처음 보는 순간"은 운에 맡기지 않는다.

각 변형이 처음 등장 가능한 스테이지의 **Wave 1에 1체를 확정 스폰**:

| 스테이지 | 보장 변형 | 적용 대상 | 의도 |
|:--------:|----------|----------|------|
| 4 | `swift` | 랜덤 적 1체 | "뭐야 빠르다!" — 속도 위협 인지 |
| 7 | `armored` | 랜덤 적 1체 | "안 죽어?!" — 화력 전략 전환 |
| 10 | `splitter` | 랜덤 적 1체 | "둘로 갈라졌어!" — 놀라움 |
| 13 | `shielded` | 랜덤 적 1체 | "앞에서 안 먹히네!" — 위치 전략 |

구현 방식:
```lua
-- GUARANTEED_VARIANTS 테이블:
-- { stage = N, variant = "name" }
-- _spawnWave()에서 wave == 1일 때 해당 스테이지 확인
-- 매칭되면 랜덤 적 1체를 해당 변형으로 강제 생성
-- 나머지 적은 기존 _pickVariant() 확률 그대로
```

#### 규칙 3: 변형 확률 스케일링

> 스테이지가 올라갈수록 변형 적 비율이 자연스럽게 증가.

현재 `_pickVariant()`의 고정 확률을 **스테이지 진행 함수**로 교체:

```lua
-- 현재 (고정):
--   swift    = 15% (stage 4+)
--   armored  = 12% (stage 7+)
--   splitter = 10% (stage 10+)
--   shielded =  8% (stage 13+)

-- 변경 (스케일링):
--   chance = baseChance × (1 + (stage - firstStage) × 0.03)
--   cap: 각 변형 최대 25%
--
-- 예시: swift (base 15%, first 4)
--   Stage 4:  15% × 1.00 = 15%
--   Stage 10: 15% × 1.18 = 17.7%
--   Stage 20: 15% × 1.48 = 22.2%
--   Stage 30: capped at 25%
--
-- 전체 변형 확률 합산 cap: 50% (절반 이상이 변형이면 변형이 "특별"이 아님)
```

#### 규칙 4: 포메이션 + 변형 조합

> 포메이션 내 적에게도 변형이 적용될 수 있도록 연결.

```lua
-- 현재: _spawnFormation()은 기본 적만 생성
-- 변경: _spawnFormation() 호출 시 variant 파라미터 전달
--   - 보장 변형 웨이브면 → 포메이션 리더(첫 번째 적)에 변형 적용
--   - 일반 웨이브면 → 각 적에 _pickVariant() 개별 적용
```

#### 전체 스테이지 체감 흐름 (1~16+)

> **왜 이 순서인가**: 재미의 4기둥이 스테이지 진행에 따라 순차적으로 자극됨.

```
[입문]   Stage 1~2   기본 적만. "이동하면 살 수 있다"
                     → 긴장감 (Tension) 최초 점화

[첫 보스] Stage 3    NULL 보스. "보스가 있구나!"
                     → 놀라움 (Wonder)

[변형 시작] Stage 4  swift 첫 등장 + 포메이션 시작
                     → 놀라움 + 긴장감 동시 상승

[중반]   Stage 5~8   테마 교대 시작. armored 등장 (Stage 7)
                     → 몰입감 (Flow) — 스테이지마다 다른 긴장의 질

[후반]   Stage 9~14  splitter(10) + shielded(13)
                     → 성취감 (Mastery) — "이것도 클리어!"

[최종]   Stage 15    OVERFLOW. 모든 것의 종합
                     → 4기둥 전부 폭발

[Endless] Stage 16+  동일 규칙 무한 반복 + 난이도 스케일링
                     → "한 판 더" 리텐션
```

## 3. 난이도 스케일링

### 파라미터 공식

| 파라미터 | 공식 | Cap |
|----------|------|-----|
| 웨이브당 적 수 (기본) | `3 + (stage-1) * 2` | 15 |
| 적 HP 배율 | `1.0 + (stage-1) * 0.15` | 없음 (무한 성장) |
| 적 이동속도 배율 | `1.0 + (stage-1) * 0.1` | 2.0x |
| 탄막 속도 배율 | `1.0 + (stage-1) * 0.05` | 1.5x |
| 웨이브 간격 | `max(1.5, 3.0 - (stage-1) * 0.3)` | 최소 1.5초 |

### 스테이지 내 웨이브 램프
- 웨이브 1: 기본 적 수의 ~50%
- 웨이브 중반: ~75%
- 최종 웨이브: 100%
- 공식: `count * (0.5 + 0.5 * wave/wavesPerStage)`

## 5. 확장 계획

### Phase 3A: 보스 스테이지 (구현 완료 ✅)

`stageManager`에서 `isBossStage` 플래그로 보스 스테이지를 감지하며, 웨이브 대신 보스 1체만 등장.
보스 매핑은 `stageData.lua`의 `BOSS_STAGES`에 정의, 보스 프리셋은 `bossDefs.lua`에 정의:

| 보스 | 스테이지 | HP | Phase | 특징 |
|------|:---:|:---:|:---:|------|
| NULL | 3 | 50 | 2 | 기본 탄막 (튜토리얼) |
| STACK | 6 | 100 | 3 | circle + aimed 조합 |
| HEAP | 9 | 160 | 3 | 텔레포트 + 점멸 경고 |
| RECURSION | 12 | 200 | 3 | 공전 이동 + 미니언 소환 |
| OVERFLOW | 15 | 300 | 4 | 전 패턴 순환 + Phase별 색상/이동 |

보스 처치 보상: 탄막 전소거 + XP 폭발 + HP/대쉬/포커스 회복

### Phase 5A.3: 적 변형 + 포메이션 (구현 완료 ✅)

| 변형 | 효과 | 첫 등장 |
|------|------|:------:|
| Swift | 속도×1.5, HP×0.8, 크기×0.8 | Stage 4 |
| Armored | HP×2.5, 속도×0.7, 크기×1.3 | Stage 7 |
| Splitter | 사망 시 미니 2마리 분열 | Stage 10 |
| Shielded | 전방 90° 피탄 무효 | Stage 13 |

포메이션 5종: Wedge(Tier1), Pincer/Triangle(Tier2), Escort/Spiral Array(Tier3)

### Phase 5A.3f: 스테이지 변형/포메이션 배치 (구현 중 🔄)

§2.1의 4가지 규칙으로 구현:
- 규칙 1: 스테이지 테마 (기동형/화력형 교대)
- 규칙 2: 변형 보장 등장 (첫 만남 확정 스폰)
- 규칙 3: 변형 확률 스케일링 (스테이지 함수)
- 규칙 4: 포메이션 + 변형 조합

### Phase 5B+: 엔드게임 (예정)
- Victory Scene (OVERFLOW 처치 후)
- Endless 모드 (Stage 16+ 보스 루프)
- DNA 시스템 (절차적 적 생성, doc/15 참조)

### Phase 5C: 커스텀 스테이지 (예정)
- JSON 기반 스테이지 로드 + 인게임 에디터

## 6. 밸런스 레퍼런스

### 세션 페이스 목표
| 구간 | 시간 | 스테이지 |
|------|------|---------|

### 레드라인 (절대 규칙)
- Stage 1~2: 즉사 금지
- 1초 이내 화면 가득 탄막 금지
- 최소 2가지 회피 경로 존재
- 적 15초 생존 → 자동 디스폰 (병목 방지)
